# Changelog — teek-ui

> **Alpha**: teek-ui is early and the API will change between minor versions.

All notable changes to teek-ui will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## Unreleased

Nothing has shipped yet - this section is a snapshot of the current API, not a running log. It gets folded/rewritten as the shape settles rather than appended to bead-by-bead.

- **Retained-mode build.** `Teek::UI.app(title:, **app_opts, &block)` builds a `Session` and yields it - Tk-free, no interpreter exists yet. `session.document` (a `Node`/`Document` tree) is constructible and traversable with no display, which is what makes UI structure testable headlessly.
- **Widgets.** `ui.<widget>` methods declare widgets into the tree. Leaf: `text_box`, `text_area`, `label`, `button`, `checkbox`, `radio`, `slider`, `dropdown`, `number_box`, `list`, `table`, `tree`, `progress`, `divider`. Containers (block-scoped children): `panel`/`box`, `group`, `canvas`, `window`. A name makes a widget addressable later via `ui[:name]`, without holding a reference.
- **Realize.** `#realize` (also called by `#run`/`#run_async`) validates the build tree, then creates the underlying `Teek::App` and walks the tree into it, idempotently and atomically - a validation failure means no interpreter is ever constructed at all, and even past that point the root window stays withdrawn until the whole tree realizes, so a mid-realize error never shows a half-built window either way (the app is destroyed and the session is left as if realize had never been called).
- **Validation.** Before any Tk call, the whole tree is checked and every problem found is raised together in one `ValidationError`, not one cryptic Tcl failure at a time: a dangling event target (naming both ends), more than one widget in the same grid cell, and (only reachable via direct tree manipulation, since the DSL itself can't produce them) grid-position intent under a non-grid parent. A named widget that's declared but never attached anywhere warns by default, or raises too under `realize(strict: true)`/`run(strict: true)`/`run_async(strict: true)`.
- **Scrolling.** `ui.scrollable(x: false, y: true) { }` auto-wires a `ttk::scrollbar` to its content - zero `-yscrollcommand`/`-xscrollcommand`/scrollbar wiring in app code. A single `list`/`text_area`/`table`/`tree`/`canvas` child is natively scrollable, so the scrollbar is wired straight to it (`-yscrollcommand`/`-xscrollcommand` pointed at the scrollbar, the scrollbar's own `-command` pointed back at `yview`/`xview`). Anything else - no children, several, or a plain container of arbitrary widgets - is wrapped in an embedded `canvas`+viewport `ttk::frame` instead (the classic Tk scrollable-frame trick), since arbitrary widgets have no scrolling protocol of their own to hook into; the viewport tracks the canvas's own width automatically unless `x:` scrolling is on, so content isn't left narrower than the visible area. Mouse-wheel scrolling works for the embedded-frame case too - `<MouseWheel>`/`<Button-4>`/`<Button-5>` all drive the same scroll regardless of which widget nested inside it the pointer happens to be over (a shared bindtag applied to the canvas, its viewport, and everything already inside them), and `Shift`+wheel scrolls horizontally when `x:` is on; native widgets already wheel-scroll through Tk's own class bindings, unchanged.
- **Layout.** `ui.column`/`ui.row` are flow containers - `gap:` (space between children), `align:` (`:start`/`:center`/`:end`/`:stretch` on the cross axis), `pad:` (margin around the whole stack). `grow: true` on any child (leaf or container) makes it consume leftover main-axis space; `ui.spacer` is a plain flexible gap (`grow: true` baked in) - the named replacement for the classic invisible spring-row trick. `ui.grid` is for the minority of screens flow doesn't fit - `g.cell(row:, col:, span:)` positions the one widget declared in its block, `g.stretch(columns:, rows:)` says which absorb leftover space. None of Tk's own geometry vocabulary (`pack`/`grid`/`sticky`/`anchor`/`rowconfigure`/`-weight`) ever appears in app code. Every other container type (`panel`/`group`/`canvas`) still just packs its children top-to-bottom with no options - overlay layout isn't built yet.
- **Handles.** One handle type across both phases - `.path`/`.configure` raise `NotRealizedError` until a widget is realized, then act on the live widget at its real, hierarchical Tk path (derived from the widget's name, nested under its parent's path).
- **Events.** `handle.on_click`/`.on_right_click` (fires on the platform's actual right-click event, whichever it is)/`.on_drag { |x, y| }` (Integer, converted through canvasx/canvasy when bound to a canvas)/`.on_key(spec)` - `spec` is a friendly symbol (`:enter`, `:escape`, `:up`, ...) or a `"Ctrl-s"`-style modifier string, resolved by `Keysyms`, which also covers the Shift-Tab/ISO_Left_Tab cross-platform gotcha. Called before realize, these queue onto the node (resolved when the whole tree realizes, including references to widgets declared later in the build); called after, they wire immediately - same handle, either phase.
- **Close handling.** `window_handle.on_close { }` fires when a `ui.window`'s close button (titlebar close box, Cmd-W, Alt-F4, ...) is pressed - teek's own default (destroy the window) only applies if nothing has claimed it, so the block decides whether the window actually closes and calls `.destroy` itself if it wants that. Same queue-before-realize/wire-immediately-after duality as every other event; rebinding replaces rather than accumulating. Raises `ArgumentError` on any non-`window` handle.
- **Escape hatch.** After realize, `session.app`/`#every`/`#after` reach the real `Teek::App` (all raise `NotRealizedError` before realize rather than queuing). During build, `session.app` doesn't exist yet, so `ui.raw { |app| ... }` is the build-time counterpart - it defers the block to realize instead of running it immediately, and (since it's a closure) can still resolve a sibling widget by name even if that sibling is declared later in the build, the same forward-reference guarantee event `target:` gets.
- **Reactive variables.** `ui.var(initial)` wraps a Tcl variable - its name is allocated at build time (no interpreter needed), the variable itself and its initial value become real at realize. Bind it to a widget with `bind: var` (translated to `-textvariable` or `-variable` depending on widget type - `text_box`/`label`/`dropdown`/`number_box` vs. `checkbox`/`slider`/`progress`); multiple bound widgets stay in sync for free via Tk's own native machinery. `var.value`/`var.value=` read/write it directly, typed to match the initial value (Integer/Float/Boolean, else String). `var.on_change { |v| }` fires with the coerced value on every change, however it happened.
- **Menus.** `ui.menu_bar { }` declares a window's menu bar (attaches to the top-level build or a `ui.window`); `.menu(label:) { }` is one recursive method for every dropdown/submenu/cascade level - no separate Tk `cascade`/`tearoff` vocabulary. Inside a `menu_bar`/`menu`/`context_menu` block, `item`/`separator`/`checkbox`/`radio` build entries - a deliberately separate vocabulary from the top-level widget DSL, so menu `checkbox`/`radio` never collide with the `ttk::checkbutton`/`ttk::radiobutton` *widgets* of the same name. `checkbox`/`radio` reuse the same `bind:` reactive-variable convention as widgets. `ui.context_menu` is a standalone popup, wired to a widget via `handle.on_right_click(menu)` (an alternative to `on_right_click`'s existing block form, not both at once). Routes entirely through teek's own `App#menu`/`App#command`, so entry-callback cleanup (including across rebuilds via `session.add`) applies automatically - no raw menu Tcl in app code.
- **Windows.** `ui.window(title:, geometry:, resizable:, modal:) { }` is a managed toplevel - it wires up title/initial geometry/resizable (a Boolean for both axes, or a `[width, height]` pair)/transient-to-whichever-window-it's-nested-under/macOS's shared-menubar quirk, and starts **withdrawn** rather than visible. `handle.show` positions it just to the right of its parent, deiconifies, raises it, and (only if declared `modal: true`) grabs input and focus; `handle.hide` releases any grab and withdraws it again. `ui.dialog(...)` is `ui.window` with `modal: true`/`resizable: false` defaults flipped for the common small-modal-window case - both still overridable. Underneath, `handle.modal(global:) { }`/`#grab_release` grab/release input directly (what `show`/`hide` call internally) - these don't queue before realize like click/key/close events do (there's nothing to grab before the window exists), so they raise `NotRealizedError` until it does; `modal`'s grab persists after a successful setup block (released explicitly, not automatically), but releases immediately if the block raises or if the window is destroyed while still grabbed. All four (`show`/`hide`/`modal`/`grab_release`) are thin delegates to base teek's `Teek::Window` - no grab/focus/safety-net logic lives in teek-ui itself - and all raise `ArgumentError` on a non-`window` handle.
- **Screens.** `ui.screens` is a push/pop stack for swapping which content is on display, working directly against ordinary DSL handles - a `ui.panel`/`ui.box`, or a `ui.window` - instead of requiring a bespoke per-screen class with its own show/hide/cleanup protocol. `.push(name, handle)` conceals whatever screen was on top (if any) and reveals the new one; `.pop` reverses that, re-revealing whatever's now on top; `.replace_current(handle)` swaps the current screen in-place without changing stack depth. `.current`/`.current_screen`/`.size`/`.active?` read the stack without mutating it. A `:window` handle is revealed/concealed through its own `.show`/`.hide` (deiconify/raise/modal, or grab-release/withdraw); anything else is packed to fill its parent, or pack-forgotten via the plain `pack`/`pack forget` primitive. Since containers pack unconditionally as soon as they realize, sibling panel screens all start visible until `ui.screens` has touched them - push every candidate once during setup, or use `ui.window` (already withdrawn by default) for screens that shouldn't show until pushed.
- **Modal stacking.** `ui.modal` is a push/pop stack for modal dialog windows, so one dialog can push another (Settings → Replay Player) with the previous one automatically re-shown once the new one is dismissed. Unlike `ui.screens`, it isn't created automatically - assign a `Teek::UI::ModalStack.new(on_enter:, on_exit:, on_focus_change: nil)` yourself, since its callbacks are mandatory and app-specific (e.g. pausing/resuming whatever's running underneath). It wraps `ui.screens` internally for the actual reveal/conceal-on-transition bookkeeping, layering on the callback lifecycle: `on_enter` fires once on the empty → non-empty transition, `on_exit` once on non-empty → empty, `on_focus_change` (optional) on every push and every pop that still leaves a dialog underneath. `.current`/`.size`/`.active?` read the stack. Push handles declared `modal: true` (`ui.dialog` already defaults to this) so `.show` actually grabs input - `ui.modal` does no grabbing of its own.
- **Dynamic UIs.** `session.add(parent_name) { }` builds a subtree with the same widget DSL and realizes it immediately as a child of an already-realized widget - "nothing happens until realize" describes the initial declaration only. Routes through the same `Teek::App#command`/leak-cleanup path the initial realize uses, so destroying an added widget reclaims its callbacks normally.
